Scenario 01

情境一：FastAPI 啟動到完成初始化

啟動的真正控制中心是 build_app_container()，負責把 runtime graph 建齊； create_app() 把 transport 組起來，再透過 lifespan() 把 container 掛進 app.state。

啟動初始化流程圖

從 uvicorn 載入 app.api.main:app 到 AppContainer 完整就緒、路由註冊完成。

重點在啟動期控制權如何一路收斂到 AppContainer，然後透過 lifespan 掛進 FastAPI。

逐步說明（共 9 步）

01
ASGI server 載入入口

執行 uv run uvicorn app.api.main:app。Python import app/api/main.py 時，module 最下方的 app = create_app() 立刻執行，容器建立不等第一個 request。

02
讀取 runtime 設定

load_runtime_config()（app/core/config.py）從環境變數組出 AppRuntimeConfig，包括 app_name、model_name、session_db_uri、toolbox_server_url、cors_allow_origins 等。這份設定是整個初始化鏈的共同輸入。

03
建立 ADK Agent

create_agent(config)（app/agent.py）做三件事：讀取 app/prompts/insurance_agent_prompt.txt、建立 ToolboxToolset(MCP)、組裝 Agent。Agent 掛入 get_user_profile_snapshot、save_user_profile、save_last_recommendation、clear_last_recommendation 四個 session tools，以及外部 MCP toolbox。

04
建立 SessionService

create_session_service(config) 依 session_db_uri scheme 決定後端：sqlite:// → SqliteSessionService，否則 → DatabaseSessionService。SQLite 路徑先經 _normalize_sqlite_db_path() 處理。

05
建立 ADK Runner

create_runner(config, agent, session_service) 初始化 Runner（app_name + agent + session_service）。後續每次 run_async() 都使用這同一個 Runner 執行。

06
封裝成 AppContainer

build_app_container() 把上述元件組成不可變的 AppContainer（config / agent / session_service / runner），route 層只需拿 container，無需了解各元件如何建立。

07
建立 FastAPI app 本體

create_app() 宣告 lifespan()（啟動時執行 app.state.container = app_container），建立 FastAPI(...)，加入 CORSMiddleware，宣告 /healthz 與 /readyz，最後 include_router(sessions)、include_router(run)。

08
Health / Readiness 端點

/healthz 只回報 app 存活；/readyz 透過 ReadinessService.collect_errors() 實際檢查 session_service 是否可取得、toolbox_server_url 是否可連通。有錯則 503。

09
Request 期依賴解析

get_container(request)（api/dependencies.py）有兩條路徑：有 request → request.app.state.container；無 request（測試情境）→ lru_cache cached container。get_runner()、get_agent()、get_session_service() 都從 container 解構而來。

控制中心 build_app_container() 決定 runtime graph；create_app() 只負責把它接進 FastAPI，兩者職責明確分離。
生命週期掛點 lifespan() 是把 container 放進 app.state 的唯一時機，確保 request 期不必重建任何 runtime 元件。
依賴回收策略 dependencies.py 的 fallback（lru_cache）讓測試與非 request 情境仍可工作，不需要 mock 整個 FastAPI stack。
Scenario 02

情境二：POST /api/agent/run 到 SSE 回傳

route 本身只負責 HTTP 驗證與 SSE 組裝；真正的流程控制在 AgentRunService.stream()，轉譯工作在 adk_event_mapper。

Request / Response 契約

AgentRunRequest（api/schemas.py）定義三個欄位：prompt: str、sessionId: str、sessionState: dict[str, str] = {}。 型別不符合 FastAPI 直接回 422，不進 route handler。 Response 是 text/event-stream，每個 frame 格式為 data: <json>\n\n。

run API 執行流程圖

從 Client 送出 POST 到 StreamingResponse 持續輸出 SSE 的完整路徑。

重點在三層協作：route（HTTP/前置驗證）→ AgentRunService（ADK 執行與 stream 生命週期）→ adk_event_mapper（event 轉 envelope）。

逐步說明（共 12 步）

01
Route 進入點 — run_agent()

收到 POST 後，FastAPI 用 AgentRunRequest 做第一層驗證（型別不符 → 422）。Route 自行做第二層：prompt.strip() / sessionId.strip() 任一為空 → 回 HTTP 400 {"error": "prompt and sessionId are required"}。

02
取得 container 與 application services

get_container(request) 取回 AppContainer，建出 SessionFacade 與 AgentRunService。取 runner 時會先嘗試 get_runner(request)，遇 TypeError 則退回 get_runner()（兼容測試 monkeypatch）。

03
確保 session 存在

await run_service.ensure_session(session_id, payload.sessionState) → SessionFacade.ensure_session() → create_session_if_missing()。先嘗試 get_session()，若不存在則 create_session() 並把 sessionState 當初始 state 寫入。失敗 → HTTP 502 JSON。

04
建立 SSE generator 並送出 meta envelope

用 payload.model_copy(update={...}) 建立 trim 後新 payload，宣告 async generator sse_generator()，回傳 StreamingResponse(media_type="text/event-stream")，headers 固定 Cache-Control: no-cache, no-transform、Connection: keep-alive。

05
AgentRunService.stream() 啟動

初始化 sequence = 0、current_text = ""、merged_state = dict(request.sessionState)，立刻 yield build_meta_envelope()（client 一收到 SSE 就先看到 transport 說明，不需等 ADK 第一個 event）。

06
包裝 user prompt 成 ADK message

iter_run_events(runner, user_id, session_id, prompt, state_delta) → build_user_message_content(prompt) 把字串包成 Content(role="user", parts=[Part(text=prompt)])，再呼叫 runner.run_async(user_id, session_id, new_message, state_delta=sessionState or None)。

07
Runner 執行 Agent 與工具

ADK Runner 執行 insurance_agent_prompt.txt 推理。需要使用者 profile → 呼叫 session_tools.py（get_user_profile_snapshot / save_user_profile / save_last_recommendation / clear_last_recommendation）直接讀寫 ADK state；需要外部資料 → 透過 ToolboxToolset 呼叫 MCP server。

08
過濾與映射 ADK events

每收到一個 event：先 is_echoed_user_input(event, prompt) 判斷是否為 ADK 回吐原文（是 → skip），sequence += 1，map_adk_event_to_envelopes(event, sequence) 轉成 envelope 陣列，merge_state_patches(merged_state, envelopes) 同步更新 state。

09
Envelope 類型對應

function_call → timeline(kind="tool-call")； function_response → timeline(kind="tool-result")； partial text → timeline(kind="stream") + message(mode="append")； final text → timeline(kind="agent") + message(mode="replace")； event.actions.state_delta → timeline(kind="state") + state(patch=...)。

10
維護 current_text 與 merged_state

message(mode="append") → 文字接在後面；message(mode="replace") → 整段覆蓋。state(patch=...) → 透過 merge_state_patches() 合入 merged_state。串流結束時可拿到最終文字與累積 state。

11
串流結束後補 done envelope

ADK event stream 結束後，重新查 session storage（SessionFacade.get_state() → get_session_state() → session_service.get_session()），把 session.state 丟進 build_public_state() 過濾 _ui_ 開頭 key，再與 fallback_state 合併，產出 build_done_envelope(final_text, state)。

12
SSE 實際輸出格式

每個 envelope 經 encode_sse_event(envelope) 轉成 data: <json>\n\n。完整 envelope 類型序列：meta → timeline / message / state（循環）→ done（或 error）。

SSE Envelope 類型說明

meta串流開始時第一個送出，描述 transport 資訊
timeline(kind="tool-call")Agent 呼叫工具時發出
timeline(kind="tool-result")工具回傳結果時發出
timeline(kind="stream")Agent 部分文字輸出
timeline(kind="agent")Agent 完整最終文字
timeline(kind="state")ADK state_delta 更新
message(mode="append")文字片段累加
message(mode="replace")整段最終文字取代
state(patch=…)session state 差分更新
done串流結束，帶最終文字與持久化 state
errorADK 執行中發生例外
前置保證 在任何 SSE 開始前，route 先 trim 輸入，並透過 ensure_session() 確保 session 存在，失敗立刻 502 JSON，不進入串流。
轉譯中心 把 ADK event 轉成前端可消化資料的核心是 adk_event_mapper，route 本身不做任何 event 解析。
收尾策略 結束時重查 session storage 而非信任記憶體 merged_state，確保 done envelope 的 state 接近實際持久化結果。
Shared Core

共用核心元件關係

兩個情境共用同一組 runtime 物件：情境一負責 build，情境二負責 consume。交會點是 AppContainer 與 get_container()。

初始化鏈與執行鏈的交會點

虛線箭頭表示 request 期透過 app.state 間接取回容器的依賴路徑。

用這張圖對照前兩張，確認 build（情境一）與 consume（情境二）的交會點。
Build（情境一）

啟動期由 build_app_container() 把所有 runtime 元件建齊，封裝成不可變的 AppContainer，再由 lifespan() 掛到 app.state。

Consume（情境二）

每個 request 透過 get_container() 取回同一份 AppContainer，解構出 runner / session_service / config，交給 AgentRunService 驅動 ADK 執行。

交會點

AppContainer 與 get_container() 是兩條流程的唯一橋樑。排查問題時，只要釐清是 build 階段失敗還是 consume 階段失敗，就可快速定位。

問題定位指南： 啟動就失敗 → app/core/config.py、app/core/container.py、app/agent.py｜ /readyz 不通 → app/application/health.py｜ /api/agent/run 前置失敗 → app/api/routes/run.py、app/api/session_service.py｜ SSE 內容不對 → app/application/agent_run_service.py、app/api/streaming.py、app/api/mappers/adk_event_mapper.py｜ state 不對 → app/tools/session_tools.py、app/api/presenters/session_presenter.py、app/domain/session_state.py
Error Paths

錯誤路徑總覽

三層錯誤處理策略，每層的回應格式不同，需分層排查。

422
Request schema 不符（FastAPI 自動處理）

欄位型別錯誤或缺少必要欄位，FastAPI 直接回 422 Unprocessable Entity，不進入 route handler。

400
Route 前置驗證失敗

trim 後 prompt 或 sessionId 為空 → HTTP 400，JSON body: {"error": "prompt and sessionId are required"}。

502
Session 建立失敗

ensure_session() 拋出例外 → HTTP 502，JSON body: {"error": "Unable to ensure session: ..."}。SSE 串流尚未開始。

SSE
ADK 執行中發生例外

AgentRunService.stream() 的 except Exception as exc → yield build_error_envelope(str(exc))。HTTP status 通常仍為 200（SSE response 已建立），但串流尾端 client 會收到 error envelope 而非 done。

注意：ADK 執行中的錯誤不會觸發 HTTP 非 2xx，因為 StreamingResponse headers 在 SSE 首個 frame 送出前就已固定。前端需主動檢查是否收到 error envelope 而非 done。
